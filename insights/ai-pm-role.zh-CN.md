## 传统 PM vs AI PM

### 传统 PM

**核心职责**：
- 需求分析和文档编写
- 项目管理和进度跟踪
- 跨部门沟通和协调
- 产品上线和运营

**工作流程**：
```
需求 → PRD → 设计 → 开发 → 测试 → 上线
```

**核心技能**：
- 需求分析
- 文档编写
- 项目管理
- 沟通协调

**工具链**：
- 文档：Word、Excel、PPT
- 设计：Figma、Sketch
- 项目管理：Jira、Trello
- 沟通：邮件、会议

---

### AI PM

**核心职责**：
- 需求分析和 AI 指令设计
- 架构设计和技术选型
- 质量判断和验证
- 快速迭代和优化

**工作流程**：
```
需求 → AI 指令 → AI 生成 → 验证 → 迭代
```

**核心技能**：
- Prompt Engineering
- Architecture Design
- Quality Assurance
- Iteration Management

**工具链**：
- AI 工具：Claude Code、Cursor
- 设计：Figma（可选）
- 项目管理：GitHub Issues
- 沟通：Slack、飞书

## AI PM 的核心能力

### 1. 架构思维

**定义**：理解系统如何工作，定义组件边界和数据流。

**关键能力**：
- 定义系统边界
- 设计组件接口
- 推理数据流
- 评估技术选型

**实践方法**：
- 先画架构图，再写代码
- 定义清晰的 API 接口
- 明确数据流向和依赖关系

**案例**：
- Ontology OS：Neo4j 是数据源，NetworkX 是计算引擎，LLM 是推理层
- Fab Agent Space：React 前端 + Mock 数据 + 组件化设计

---

### 2. Prompt Engineering

**定义**：设计给 AI 的指令，让 AI 理解你的意图并产出高质量结果。

**关键能力**：
- 把需求翻译成 AI 能理解的指令
- 设计清晰的约束和验收标准
- 处理边界情况和错误

**实践方法**：
- 用结构化的方式描述需求
- 提供具体的示例和反例
- 明确技术约束和限制

**Prompt 设计原则**：
1. **清晰**：用简洁的语言描述需求
2. **具体**：提供具体的示例和约束
3. **完整**：覆盖所有边界情况
4. **可验证**：定义明确的验收标准

---

### 3. 质量判断

**定义**：判断 AI 产出的质量，识别潜在的问题和风险。

**关键能力**：
- 识别代码质量问题
- 发现架构设计缺陷
- 评估性能和安全风险
- 决定是否需要人工干预

**实践方法**：
- 建立质量检查清单
- 测试关键功能和边界情况
- 验证架构决策的合理性

**质量检查维度**：
- 功能正确性
- 边界处理
- 性能表现
- 安全性

---

### 4. 快速验证

**定义**：快速验证需求和假设，用最小成本验证想法。

**关键能力**：
- 设计最小可行产品（MVP）
- 快速生成原型
- 收集用户反馈
- 迭代优化

**实践方法**：
- 用 AI 快速生成原型
- 让用户试用并反馈
- 基于反馈快速迭代
- 验证核心假设

**验证清单**：
- [ ] 核心功能是否解决用户问题
- [ ] 交互流程是否顺畅
- [ ] 用户是否愿意使用
- [ ] 是否有商业价值

## AI PM 的工作流

### 阶段 1：需求分析（10 min）

**输入**：模糊的需求描述

**输出**：结构化的需求文档

**关键活动**：
- 理解业务背景
- 明确目标用户
- 定义核心问题
- 确定优先级

**工具**：Markdown

---

### 阶段 2：架构设计（20 min）

**输入**：结构化的需求文档

**输出**：系统架构和 API 设计

**关键活动**：
- 设计系统架构
- 定义组件边界
- 设计 API 接口
- 选择技术栈

**工具**：Markdown、Excalidraw

---

### 阶段 3：AI 生成（30 min）

**输入**：系统架构和 API 设计

**输出**：可运行的原型

**关键活动**：
- 编写 AI 指令（Prompt）
- 生成代码
- 验证功能
- 修复问题

**工具**：Claude Code、Cursor

---

### 阶段 4：验证迭代（按需）

**输入**：用户反馈

**输出**：优化后的版本

**关键活动**：
- 收集用户反馈
- 分析问题
- 优化 Prompt
- 重新生成

**工具**：Claude Code、Cursor

---

## AI PM 的挑战

### 1. 技术理解

**挑战**：需要理解技术架构和实现细节

**应对方法**：
- 学习基础的技术知识
- 与工程师沟通交流
- 通过实践加深理解

**案例**：
- Ontology OS：通过逆向工程 Palantir AIP 理解本体系统
- Fab Agent Space：通过构建原型理解 React 架构

---

### 2. 质量把控

**挑战**：AI 生成的代码质量不稳定

**应对方法**：
- 建立质量检查清单
- 测试关键功能和边界情况
- 验证架构决策的合理性

**质量检查维度**：
- 功能正确性
- 边界处理
- 性能表现
- 安全性

---

### 3. 沟通协作

**挑战**：需要与工程师、设计师、用户等多方沟通

**应对方法**：
- 用原型代替文档
- 用演示代替会议
- 用数据说话

**沟通技巧**：
- 用原型展示想法
- 用数据支持决策
- 用反馈验证假设

## AI PM 的成长路径

### 阶段 1：学习基础（0-3 个月）

**目标**：掌握 AI 工具和基础技能

**学习内容**：
- Prompt Engineering 基础
- React + Vite + Tailwind CSS
- 版本控制（Git）
- 基础的架构设计

**实践项目**：
- 简单的 CRUD 应用
- 静态网站
- 数据可视化

---

### 阶段 2：实践应用（3-6 个月）

**目标**：在实际项目中应用 AI 工具

**学习内容**：
- 复杂的架构设计
- 性能优化
- 用户体验设计
- 产品思维

**实践项目**：
- 内部工具
- MVP 产品
- 技术验证

---

### 阶段 3：深度实践（6-12 个月）

**目标**：成为 AI PM 专家

**学习内容**：
- 本体论思维
- 系统架构设计
- 产品战略
- 团队管理

**实践项目**：
- 核心产品
- 技术平台
- 团队建设
