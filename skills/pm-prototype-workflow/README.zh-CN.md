    23|# PM Prototype Workflow — AI Native 产品原型开发工作流
    24|
    25|> 从产品需求到可交互原型的 AI Native 工作流
    26|> 核心理念：PRD 不是文档，是给 AI 的指令集
    27|
    28|---
    29|
    30|## 适用场景
    31|
    32|- 新产品/功能的快速原型验证
    33|- 竞品分析后的功能复刻
    34|- 用户调研后的需求可视化
    35|- 内部汇报/演示用的 Demo
    36|- 技术方案验证（Vibe Coding）
    37|
    38|---
    39|
    40|## 工作流总览
    41|
    42|```
    43|需求输入 → AI 生成 PRD → AI 生成原型 → 快速迭代 → 交付验证
    44|   ↓           ↓              ↓            ↓          ↓
    45| 10 min     20 min         30 min       按需       即时
    46|```
    47|

    49|
    50|## 第一阶段：需求输入（10 min）
    51|
    52|### 输入模板
    53|
    54|```markdown
    55|## 需求背景
    56|- 目标用户：[谁会用这个功能？]
    57|- 核心问题：[解决什么问题？]
    58|- 业务价值：[为什么要做？]
    59|
    60|## 参考竞品
    61|- 竞品 A：[名称] - [参考点]
    62|- 竞品 B：[名称] - [参考点]
    63|
    64|## 核心功能（MoSCoW）
    65|### Must Have
    66|- [ ] 功能 1：[描述]
    67|- [ ] 功能 2：[描述]
    68|
    69|### Should Have
    70|- [ ] 功能 3：[描述]
    71|
    72|### Could Have
    73|- [ ] 功能 4：[描述]
    74|
    75|## UI/UX 参考
    76|- 设计风格：[极简/工业/现代/...]
    77|- 参考链接：[Dribbble/Behance/竞品截图]
    78|- 交互模式：[单页/多页/抽屉/弹窗/...]
    79|```
    80|
    81|### 示例：Fab Agent Space 需求
    82|
    83|```markdown
    84|## 需求背景
    85|- 目标用户：半导体 FAB 工程师
    86|- 核心问题：工程师需要在多个系统间切换查询信息
    87|- 业务价值：提升工程师效率，减少停机时间
    88|
    89|## 参考竞品
    90|- 竞品 A：Palantir AIP - 本体驱动的智能体界面
    91|- 竞品 B：ServiceNow - 任务工单系统
    92|
    93|## 核心功能（MoSCoW）
    94|### Must Have
    95|- [ ] 仪表盘：展示关键指标和待办任务
    96|- [ ] 工作台：工程师的主要工作界面
    97|- [ ] 任务大厅：任务列表和筛选
    98|
    99|### Should Have
   100|- [ ] Agent 构建器：可视化配置 Agent
   101|- [ ] 专家中心：专家知识库
   102|
   103|### Could Have
   104|- [ ] Copilot：AI 辅助助手
   105|
   106|## UI/UX 参考
   107|- 设计风格：暗黑工业风（参考 Palantir）
   108|- 参考链接：[截图/链接]
   109|- 交互模式：左侧导航 + 主内容区
   110|```

   114|## 第二阶段：AI 生成 PRD（20 min）
   115|
   116|### Prompt 模板
   117|
   118|```markdown
   119|你是一个资深产品经理，擅长将模糊需求转化为结构化 PRD。
   120|
   121|基于以下需求，生成一份面向 AI 编码工具（Claude Code）的 PRD。
   122|
   123|要求：
   124|1. 不要写传统的文档式 PRD，要写成给 AI 的指令集
   125|2. 每个功能点都要有明确的验收标准
   126|3. 技术栈固定：React + Vite + Tailwind CSS
   127|4. 组件命名要清晰，便于 AI 理解
   128|5. 交互逻辑要具体，不要模糊描述
   129|
   130|需求输入：
   131|[粘贴第一阶段的需求]
   132|
   133|输出格式：
   134|1. 产品概述（一段话）
   135|2. 核心功能清单（表格）
   136|3. 页面结构（组件树）
   137|4. 交互流程（状态机）
   138|5. 技术约束（明确边界）
   139|```
   140|
   141|### PRD 输出模板
   142|
   143|```markdown
   144|# [产品名称] PRD
   145|
   146|## 产品概述
   147|[一段话描述产品定位、目标用户、核心价值]
   148|
   149|## 核心功能清单
   150|
   151|| 功能模块 | 组件名 | 优先级 | 验收标准 |
   152||----------|--------|--------|----------|
   153|| 仪表盘 | Dashboard | P0 | 展示 3 个核心指标卡片 |
   154|| 工作台 | Workbench | P0 | 支持拖拽排列工作区 |
   155|| 任务大厅 | TaskHall | P1 | 支持筛选和排序 |
   156|
   157|## 页面结构
   158|
   159|```
   160|App
   161|├── Sidebar（左侧导航）
   162|│   ├── Logo
   163|│   ├── NavMenu
   164|│   └── UserProfile
   165|├── Dashboard（仪表盘）
   166|│   ├── MetricCards
   167|│   ├── RecentTasks
   168|│   └── QuickActions
   169|├── Workbench（工作台）
   170|│   ├── TaskPanel
   171|│   ├── AgentChat
   172|│   └── ContextPanel
   173|└── TaskHall（任务大厅）
   174|    ├── FilterBar
   175|    ├── TaskList
   176|    └── TaskDetail
   177|```
   178|
   179|## 交互流程
   180|
   181|### 状态机：任务状态
   182|```
   183|[待处理] → [进行中] → [已完成]
   184|    ↓          ↓          ↓
   185|  [暂停]    [阻塞]     [归档]
   186|```
   187|
   188|### 交互细节
   189|- 点击任务卡片 → 右侧抽屉展示详情
   190|- 拖拽任务 → 改变状态
   191|- 双击任务 → 进入编辑模式
   192|
   193|## 技术约束
   194|- 框架：React 19 + Vite 8 + Tailwind CSS 4
   195|- 状态管理：useState + useContext（无 Redux）
   196|- 路由：React Router v6
   197|- 图标：Lucide React
   198|- 样式：Tailwind CSS（无 CSS-in-JS）
   199|- 端口：5175
   200|- 部署：GitHub Pages（base: './'）
   201|```

   205|## 第三阶段：AI 生成原型（30 min）
   206|
   207|### Prompt 模板
   208|
   209|```markdown
   210|你是一个资深前端工程师，擅长用 React + Vite + Tailwind CSS 快速构建原型。
   211|
   212|基于以下 PRD，生成一个可运行的原型项目。
   213|
   214|要求：
   215|1. 使用 React 19 + Vite 8 + Tailwind CSS 4
   216|2. 所有组件都要是 functional component + hooks
   217|3. 样式全部用 Tailwind CSS，不要写自定义 CSS
   218|4. 使用 Lucide React 图标
   219|5. 数据用 mock 数据，不要调用真实 API
   220|6. 交互要完整（点击、筛选、跳转等）
   221|7. 响应式设计（支持桌面端）
   222|
   223|PRD：
   224|[粘贴第二阶段的 PRD]
   225|
   226|输出：
   227|1. 项目初始化命令
   228|2. 目录结构
   229|3. 核心组件代码
   230|4. Mock 数据
   231|5. 启动命令
   232|```
   233|
   234|### 生成流程
   235|
   236|```bash
   237|# 1. 初始化项目
   238|npm create vite@latest my-prototype -- --template react
   239|cd my-prototype
   240|npm install
   241|
   242|# 2. 安装依赖
   243|npm install tailwindcss @tailwindcss/vite lucide-react react-router-dom
   244|
   245|# 3. 配置 Tailwind CSS（在 vite.config.js 中）
   246|# 4. 创建目录结构
   247|mkdir -p src/components src/lib src/data
   248|
   249|# 5. 让 Claude Code 生成组件
   250|# （基于 PRD 逐个生成）
   251|```
   252|
   253|### Claude Code 使用技巧
   254|
   255|```bash
   256|# 启动 Claude Code
   257|claude
   258|
   259|# 然后用自然语言描述需求
   260|"根据 PRD 生成 Dashboard 组件，包含 3 个指标卡片"
   261|
   262|"生成 TaskHall 组件，支持按状态筛选任务"
   263|
   264|"添加左侧导航栏，支持路由跳转"
   265|
   266|"把所有 mock 数据放到 src/data/mock.js"
   267|```

   271|## 第四阶段：快速迭代（按需）
   272|
   273|### 迭代 Prompt 模板
   274|
   275|```markdown
   276|基于当前原型，进行以下修改：
   277|
   278|1. [修改点 1]：[具体描述]
   279|2. [修改点 2]：[具体描述]
   280|
   281|要求：
   282|- 保持现有架构不变
   283|- 只修改指定部分
   284|- 确保其他功能不受影响
   285|```
   286|
   287|### 常见迭代场景
   288|
   289|**场景 1：UI 调整**
   290|```markdown
   291|把 Dashboard 的指标卡片从 3 列改为 4 列，增加一个"待办任务"卡片
   292|```
   293|
   294|**场景 2：交互优化**
   295|```markdown
   296|点击任务卡片时，右侧弹出抽屉展示详情，而不是跳转新页面
   297|```
   298|
   299|**场景 3：功能新增**
   300|```markdown
   301|在 TaskHall 添加"创建任务"按钮，点击后弹出模态框
   302|```
   303|
   304|**场景 4：样式调整**
   305|```markdown
   306|整体色调从暗黑改为浅色，但保持工业风设计语言
   307|```

   363|## 最佳实践
   364|
   365|### 1. PRD 要写给 AI，不是写给人
   366|
   367|```markdown
   368|❌ 传统 PRD：系统应支持任务管理功能
   369|✅ AI PRD：创建 TaskHall 组件，包含 TaskList 和 TaskDetail 两个子组件
   370|```
   371|
   372|### 2. 组件命名要清晰
   373|
   374|```markdown
   375|❌ 模糊命名：Card、List、Panel
   376|✅ 清晰命名：MetricCard、TaskList、ContextPanel
   377|```
   378|
   379|### 3. 交互描述要具体
   380|
   381|```markdown
   382|❌ 模糊描述：用户可以筛选任务
   383|✅ 具体描述：顶部 FilterBar 包含状态下拉（全部/待处理/进行中/已完成）和搜索框
   384|```
   385|
   386|### 4. 技术约束要明确
   387|
   388|```markdown
   389|❌ 模糊约束：使用现代前端框架
   390|✅ 明确约束：React 19 + Vite 8 + Tailwind CSS 4，无 Redux
   391|```
   392|
   393|### 5. Mock 数据要真实
   394|
   395|```markdown
   396|❌ 随意数据：{ name: "test" }
   397|✅ 真实数据：{ id: "TASK-001", title: "拉晶断苞分析", status: "进行中", assignee: "张工" }
   398|```

   431|## 进阶用法
   432|
   433|### 1. 竞品截图 → 原型
   434|
   435|```markdown
   436|我有一张竞品截图（附图），帮我：
   437|1. 分析页面结构和组件
   438|2. 用 React + Tailwind CSS 复刻
   439|3. 保持交互逻辑一致
   440|```
   441|
   442|### 2. 设计稿 → 原型
   443|
   444|```markdown
   445|我有一个 Figma 设计稿（附链接），帮我：
   446|1. 提取设计规范（颜色、字体、间距）
   447|2. 生成对应的 Tailwind 配置
   448|3. 逐个组件实现
   449|```
   450|
   451|### 3. 用户反馈 → 迭代
   452|
   453|```markdown
   454|用户反馈：
   455|- 任务列表加载太慢
   456|- 筛选功能不够明显
   457|- 希望有快捷操作
   458|
   459|帮我优化：
   460|1. 添加加载状态
   461|2. 突出筛选栏
   462|3. 增加快捷操作按钮
   463|```
