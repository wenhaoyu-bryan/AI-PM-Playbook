# 需求拆解框架

**[English](skills/requirements-breakdown/README.md)** | **中文**


> 从模糊需求到结构化任务
> MoSCoW 优先级排序 + 功能分解树

---

## 什么是需求拆解？

**定义**：把模糊的需求描述转化为结构化的、可执行的任务列表。

**核心价值**：
- 把模糊变清晰
- 把大任务变小任务
- 把不确定变确定

**适用场景**：
- 新产品/功能的需求分析
- 竞品分析后的功能规划
- 用户调研后的需求整理
- 技术方案验证前的需求定义

---

## 需求拆解的流程

```
模糊需求 → 需求澄清 → 功能分解 → 优先级排序 → 任务列表
   ↓           ↓          ↓            ↓          ↓
 10 min     20 min     30 min       10 min      10 min
```

## 第三阶段：优先级排序（10 min）

### MoSCoW 方法

**Must Have（必须有）**：
- 没有这个功能，产品无法使用
- 核心功能，不可或缺
- 用户的刚需

**Should Have（应该有）**：
- 重要但不紧急
- 提升用户体验
- 可以延后但不能没有

**Could Have（可以有）**：
- 锦上添花
- 提升竞争力
- 有资源就做

**Won't Have（不会有）**：
- 明确不做
- 超出范围
- 资源不允许

### 优先级排序模板

```markdown
## 优先级排序

### Must Have（必须有）
- [ ] 功能点 1：[描述] - [原因]
- [ ] 功能点 2：[描述] - [原因]
- [ ] 功能点 3：[描述] - [原因]

### Should Have（应该有）
- [ ] 功能点 4：[描述] - [原因]
- [ ] 功能点 5：[描述] - [原因]

### Could Have（可以有）
- [ ] 功能点 6：[描述] - [原因]
- [ ] 功能点 7：[描述] - [原因]

### Won't Have（不会有）
- [ ] 功能点 8：[描述] - [原因]
```

### 示例：Fab Agent Space 优先级排序

```markdown
## 优先级排序

### Must Have（必须有）
- [ ] Dashboard：展示关键指标 - 工程师需要快速了解工作状态
- [ ] TaskHall：任务列表和筛选 - 工程师需要查看和管理任务
- [ ] Workbench：工作台界面 - 工程师的主要工作界面

### Should Have（应该有）
- [ ] AgentBuilder：Agent 构建器 - 支持自定义 Agent
- [ ] ExpertCenter：专家中心 - 提供专家知识库

### Could Have（可以有）
- [ ] Copilot：AI 辅助助手 - 提升效率
- [ ] SkillCenter：技能中心 - 管理 Agent 技能

### Won't Have（不会有）
- [ ] 后端 API：暂时用 Mock 数据 - 快速验证需求
- [ ] 用户认证：暂时不需要 - 内部演示用
```

## 第四阶段：任务列表（10 min）

### 任务拆解原则

**1. 可执行**：
- 每个任务都能直接执行
- 不需要再拆解
- 有明确的输入和输出

**2. 可验证**：
- 每个任务都有验收标准
- 可以判断是否完成
- 可以测试功能正确性

**3. 可估时**：
- 每个任务都能估算时间
- 时间在 1-4 小时之间
- 不会太大也不会太小

### 任务列表模板

```markdown
## 任务列表

### 任务 1：[任务名称]

- [ ] 标准 1
```

## 需求拆解的最佳实践

### 1. 需求要具体

**错误做法**：
- 系统应支持任务管理功能
- 用户可以查看任务
- 提供搜索功能

**正确做法**：
- 创建 TaskHall 组件，包含 TaskList 和 TaskDetail 两个子组件
- 任务列表支持状态下拉筛选（全部/待处理/进行中/已完成）
- 搜索框支持按任务标题搜索

---

### 2. 功能要分解

**错误做法**：
- 做一个任务管理系统
- 包含任务列表、任务详情、任务编辑

**正确做法**：
- 功能模块 1：TaskHall（任务大厅）
  - 功能点 1.1：任务列表展示
  - 功能点 1.2：任务筛选
  - 功能点 1.3：任务详情抽屉
- 功能模块 2：TaskEditor（任务编辑器）
  - 功能点 2.1：任务表单
  - 功能点 2.2：任务保存
  - 功能点 2.3：任务取消

---

### 3. 优先级要明确

**错误做法**：
- 所有功能都很重要
- 先做简单的
- 老板说做什么就做什么

**正确做法**：
- 用 MoSCoW 方法排序
- Must Have 是核心功能
- Should Have 是重要功能
- Could Have 是锦上添花

---

### 4. 任务要可执行

**错误做法**：
- 做一个仪表盘
- 实现任务管理功能
- 添加搜索功能

**正确做法**：
- 创建 Dashboard 组件，展示 3 个指标卡片
- 创建 TaskHall 组件，支持状态下拉筛选
- 添加搜索框，支持按任务标题搜索

## 需求拆解的常见错误

### 1. 需求太模糊

**错误**：系统应支持任务管理功能

**正确**：创建 TaskHall 组件，包含 TaskList 和 TaskDetail 两个子组件

---

### 2. 功能太大

**错误**：做一个任务管理系统

**正确**：
- 功能模块 1：TaskHall（任务大厅）
- 功能模块 2：TaskEditor（任务编辑器）
- 功能模块 3：TaskFilter（任务筛选）

---

### 3. 优先级不明确

**错误**：所有功能都很重要

**正确**：
- Must Have：Dashboard、TaskHall、Workbench
- Should Have：AgentBuilder、ExpertCenter
- Could Have：Copilot、SkillCenter

**错误**：做一个仪表盘
